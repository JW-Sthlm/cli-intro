Copilot CLI in Action

Live demo script for leadership team. No slides. Terminal + browser + talking. Copyable prompts on the side.

Format: You talk while you type. The CLI output IS the presentation. Keep the terminal large-font, dark theme. Have the browser ready with the live site and the repo open in separate tabs.
Pre-staging: Before the call, launch CLI inside the partner library repo: cd ~/projects/agentic-ai-partner-library && copilot. Have the live site and repo open in Edge tabs. Test that PMX/GitHub MCP work. Do a fresh az login.
Context — what you're looking at 0:00 – 1:30

DO: Show the live site briefly in browser. Then switch to terminal.

I run a curated content library for the Agentic AI partner community. 14 Microsoft sources scanned daily by agents. Domain experts review. Agents publish. I built and operate this in Copilot CLI — no code. Let me show you how a real cycle works.

Landing message: This is not a toy demo. This is a production system. Agents run overnight. Experts review in the morning. Content goes live by lunch.
The trigger — what agents found 1:30 – 4:00

DO: In CLI, ask about recent activity.

What happened in this repo this week? Show me recent issues and PRs — who created them, what were they about?

Let Copilot answer. Point out: some issues were created by github-actions (the research agent), some PRs by copilot-swe-agent, some by team members like Gilles. The system has multiple actors — agents and humans — working together.

DO: Drill into an agent-created issue.

Show me issue #50 — the one about the MAF evaluation suite. Who created it, what happened, and how was it resolved?

This is the flow: an agent scanned sources, found a relevant new repo, created an issue, tagged it for the right domain expert. The expert reviewed it, approved it, and assigned Copilot to publish. Copilot created PR #49, which added it to the site.

Message to land: Discovery, triage, and publishing — all automated. The human's job is judgment: is this relevant? Is it good enough? Three clicks from the expert. Everything else runs.
The team — who does what 4:00 – 6:00

DO: Show the team structure.

Show me the team.yml file — who are the domain experts and what areas do they cover?

Six people. Each owns a solution area. The agent knows who to assign content to based on the topic. Kiko gets CAIP content. Lorette gets Copilot Studio. Peter gets Dynamics. The routing is defined here — in a YAML file, not in code.

DO: Then show the sources.

How many sources do we scan? What types?

14 sources. RSS feeds, GitHub releases, What's-New pages. The research agent checks them daily. New content becomes issues, issues become reviews, reviews become published entries.

Message to land: This is not one person doing everything. It's a system with defined roles — some human, some agent. The configuration is readable, shareable, version-controlled.
Live interaction — ask something real 6:00 – 9:00

DO: Do something live that shows CLI working across tools.

Look at the open issues in this repo. Are there any that need my attention? Summarize what's pending and suggest priorities.

Point at the output. CLI reads the GitHub issues, understands the context, and prioritizes. Not just listing — analyzing.

DO: Then cross-reference with another source.

Check if any of the broken link issues overlap with content that was recently added. Are we creating problems with our own PRs?

This is the delta vs M365 Copilot. It's connecting GitHub issues with PR history — two different data sources — and finding a pattern. M365 Copilot can't do this. It doesn't have the connectors.

Fallback: If CLI is slow or gives a partial answer, just narrate what it's doing: "It's reading the issues, cross-referencing with recent PRs — this is the kind of multi-source analysis that would take me 20 minutes manually."
The bridge — this scales 9:00 – 12:00

This is one workflow. A content library. But the same pattern works for any repeatable partner work that spans multiple sources and multiple steps.

Partner briefings — pull PMX, MSX, email, synthesize into talking points. Workshop follow-ups — turn raw notes into phased action plans. Opportunity analysis — cross-reference pipeline data with market research. Same tool. Same connectors. Different workflow.

We're running an intro session for the team to get everyone started. 60 minutes, hands-on, they build real workflows with their own partner data. No coding. They walk away with a working setup they can keep using.

DO: Optional — if time allows, do one quick partner-work prompt.

Show me my PMX projects that are missing outcomes. Which ones should I fix first?

That just hit Dynamics 365 live. From the terminal. Plain English. This is what the team will learn to do in the intro session.

Message to land: I built and operate a production system in CLI. The same tool connects to PMX, MSX, M365, GitHub. The intro session gives everyone a working setup and tested patterns. This is us being customer zero for AI-first workflows.
